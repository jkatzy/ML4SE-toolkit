## Confirmed Bugs and Policy Questions

### cweb

- Family: `cweb_style`
- Fixture file: `tests/fixtures/comment_languages/cweb.code`
- Status: confirmed-bug
- Fixture edit: kept only parseable one-line
  `@q source-only af_cweb_control keeps // and % markers @>`
- Extraction entry point: `CommentQuery("cweb")`
- Sanitizer entry point: `CommentSanitizer("cweb")`
- Source/provenance:
  `https://mirror.gutenberg-asso.fr/tex.loria.fr/litte/cwebman.pdf`
  states that CWEB control texts end at the next `@>` and the closing `@>` must
  be on the same line as the control text start.

Finding 1: fixture generator emits an invalid multiline CWEB `@q...@>` expected
match.

Input generated by `scripts/build_comment_language_fixtures.py`:

```text
@q source-only
 * star_doc_1 Description of what the method does.
 *
 * @param input Description of parameter.
 * @return Description of return value.
 * @throws Exception Description of exception.
@>
```

Actual behavior:

```text
parsed CWEB matches in the fixture:
@q source-only fixture_1@>
@q source-only inline_block_1@>
@q source-only af_cweb_control keeps // and % markers @>
```

Expected behavior:

```text
The fixture builder should not generate star-prefixed multiline doc-block cases
for line-bound CWEB @q control text, or the registry example should opt out of
that generated block shape.
```

Why it matters: this pre-existing generated expectation makes the canonical
fixture test fail before and after breaker fixture additions.

Finding 2: sanitizer only strips CWEB `@q` comments when the body starts with
the seeded literal `source-only`.

Findings-only input:

```text
@q af_cweb_generic_sanitize_probe @>
```

Raw extracted match:

```text
@q af_cweb_generic_sanitize_probe @>
```

Actual sanitized output:

```text
@q af_cweb_generic_sanitize_probe @>
```

Expected sanitized output:

```text
af_cweb_generic_sanitize_probe
```

Removal contract: strip the CWEB `@q` opener and `@>` closer as syntax.
Preservation contract: preserve the arbitrary control text body.
Paired preservation testcase: the kept fixture case
`@q source-only af_cweb_control keeps // and % markers @>` sanitizes to
`af_cweb_control keeps // and % markers`, preserving the `//` and `%` body
markers.

### cue_sheet

- Family: `cue_sheet_style`
- Fixture file: `tests/fixtures/comment_languages/cue_sheet.code`
- Status: needs-policy-decision
- Fixture edit: kept `rem` case-insensitive, indented REM fixture
  `af_cue_sheet_lower_rem`
- Extraction entry point: `CommentQuery("cue_sheet")`
- Sanitizer entry point: `CommentSanitizer("cue_sheet")`
- Source/provenance:
  `https://www.gnu.org/software/ccd2cue/manual/html_node/REM-_0028CUE-Command_0029.html`
  documents standard line-ending `REM` comments.
  `https://github.com/libyal/libodraw/blob/main/documentation/CUE%20sheet%20format.asciidoc`
  documents observed CUE comments using `;` and cdrdao-style `//`.

Findings-only probes:

```text
; af_cue_semicolon_probe
TRACK 01 AUDIO
```

```text
// af_cue_slash_probe
TRACK 01 AUDIO
```

Actual behavior:

```text
[]
```

Expected behavior:

```text
Policy-dependent. The current registry intentionally supports only standard
line-anchored REM remarks, but public CUE documentation records semicolon and
// variants in the wild.
```

Removal contract: if these variants are accepted later, strip only the chosen
line-comment opener.
Preservation contract: keep body text such as URLs, `# Heading`, or `//` when
it is not the opener. Do not add these variants to fixtures until policy is
settled.

### csound, csound_document, csound_score, cycript, eagle, ec, edje_data_collection

- Families: `csound_style`, `csound_document_style`, `csound_score_style`,
  `cycript_style`, `eagle_style`, `ec_style`,
  `edje_data_collection_style`
- Status: confirmed-bug for sanitizer under-stripping in a shared C-style
  block shape
- Fixture edit: kept opener-on-own-line doc-gutter cases because they parse and
  sanitize correctly. The mixed opener-line bug probe was not left in fixtures.
- Extraction entry point: `CommentQuery`
- Sanitizer entry point: `CommentSanitizer`
- Source/provenance:
  `https://csound.com/docs/manual/OrchTop.html`,
  `https://csound.com/docs/manual/ScoreStatements.html`, and
  `https://web.mit.edu/xavid/arch/i386_rhel4/help/133.htm`
  document C-style `/* ... */` blocks for representative languages; EAGLE also
  documents that such comments stop at the first `*/` and do not nest.

Findings-only representative input:

```text
/* af_cycript_block has // marker
 * * bullet must remain
 * # Heading must remain
 */
```

Raw extracted match:

```text
/* af_cycript_block has // marker
 * * bullet must remain
 * # Heading must remain
 */
```

Actual sanitized output:

```text
af_cycript_block has // marker
 * * bullet must remain
 * # Heading must remain
```

Expected sanitized output:

```text
af_cycript_block has // marker
* bullet must remain
# Heading must remain
```

Removal contract: remove the decorative leading doc-gutter `*` and alignment
spaces from following lines even when the block opener line also contains body
text.
Preservation contract: preserve the meaningful Markdown/list `* bullet`, the
Markdown `# Heading`, and the literal `//` text in the first line body.
Paired preservation testcase: opener-on-own-line fixture cases such as
`af_cycript_doc_gutter`, `af_eagle_doc_gutter`, and `af_csound_doc_gutter`
sanitize correctly to bodies that preserve `* bullet` and `# Heading`.
